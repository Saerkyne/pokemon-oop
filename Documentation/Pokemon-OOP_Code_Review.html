Part 1: Bugs (Things That Are Broken)

BUG 1 — Critical Hit Logic Is Inverted

File: Attack.java, lines 28–35

The condition if (chance > threshold) means slower Pokémon crit more often — the opposite of the intended behavior. With threshold = speed / 2:

SpeedThresholdCrit WhenCrit Rate
200100random > 100~60%
5025random > 25~90%

Fix: Change > to <:

if (chance < threshold) {

Now faster Pokémon crit more often (higher speed → higher threshold → more values qualify).

BUG 2 — Hardcoded Attack/Defense in Damage Formula

File: Attack.java, lines 68–69

int attackStat = 168; //attacker.getAttackStatForMove(move);
int defenseStat = 218; //defender.getDefenseStatForMove(move);

Every attack deals the same damage regardless of actual stats. Uncomment the real calls — but first see Bug 3, because the methods they call also have a problem.

BUG 3 — getAttackStatForMove / getDefenseStatForMove Return Base Stats, Not Calculated Stats

File: Pokemon.java, lines 208–223

These methods call getAttackBaseStat() / getDefenseBaseStat() etc., which are the raw base stats (e.g. 49 for Bulbasaur's Attack) — ignoring level, IVs, EVs, and nature entirely. When you uncomment the calls in Attack.java, they need to use the calculated stats instead:

// Before (wrong):
int physAttack = getAttackBaseStat();

// After (correct):
int physAttack = getCurrentAttack();

Same fix for all four branches (physical attack, special attack, physical defense, special defense).

BUG 4 — System.console() Returns null in IDEs

Files: Pokemon.java (line 107) and LearnsetEntry.java (line 40)

System.console() returns null when running from VS Code, IntelliJ, or any non-terminal launcher, causing an immediate NullPointerException. Use a Scanner wrapping System.in instead:

Scanner scanner = new Scanner(System.in);
String response = scanner.nextLine();

File: moves/Disable.java

Status moves should have 0 power. If Disable goes through the damage formula, it will deal damage despite being categorized as Status. Change power to 0.

BUG 8 — teachFromLearnset Will NPE if Pokémon is null

The null check on p assigns null to catalog, then immediately calls catalog.size(). Either return early or throw with a meaningful message:

BUG 9 — Inconsistent null vs "None" for Secondary Type

Some Pokémon use "None" while others use null for no secondary type. calculateEffectiveness handles both, but any future code that does .equals() on the secondary type without a null check will crash.

ConventionSpecies Using It
nullSquirtle, Charmeleon, Wartortle, Blastoise, Mewtwo, Eevee
"None"Charmander, Pikachu

Fix: Pick one convention and apply it everywhere. null is cleaner (then guard with null checks), or "None" is more explicit.

Part 3: Efficiency Improvements (Detailed)

1. TypeChart — Rebuilding the Lookup Map Every Call

File: TypeChart.java, getEffectiveness() method

Current code:

public float getEffectiveness(String moveType, String pokemonType) {
    Dictionary<String, Integer> typeIndices = new Hashtable<>();
    typeIndices.put("Normal", 0);
    typeIndices.put("Fire", 1);
    // ... 18 entries rebuilt every single call
}

Every time you check type effectiveness (multiple times per attack), this allocates a new Hashtable and inserts 18 entries. In a battle loop, this runs hundreds of times.

Better approach — build the map once as a static final field:

private static final Map<String, Integer> TYPE_INDICES = new HashMap<>();
static {
    TYPE_INDICES.put("Normal", 0);
    TYPE_INDICES.put("Fire", 1);
    // ...all 18 types
}

public float getEffectiveness(String moveType, String pokemonType) {
    int moveIndex = TYPE_INDICES.get(moveType);
    int pokemonIndex = TYPE_INDICES.get(pokemonType);
    return typeChart[moveIndex][pokemonIndex];
}

This also replaces the deprecated Dictionary/Hashtable with Map/HashMap.

2. Attack.java — Creating a New Random and TypeChart Per Instance

File: Attack.java

Current code:

TypeChart typeChart = new TypeChart();  // new instance per Attack object

public int randomInt(int min, int max) {
    Random rand = new Random();  // new Random per call
    return rand.nextInt((max - min) + 1) + min;
}

new Random() called rapidly can produce identical seeds (same results). And a new TypeChart (which builds the 18×18 matrix) per Attack instance is wasteful.

Better approach:

private static final Random RNG = new Random();
private static final TypeChart TYPE_CHART = new TypeChart();

public int randomInt(int min, int max) {
    return RNG.nextInt((max - min) + 1) + min;
}

One Random, one TypeChart, shared across all Attack usages.

3. App.createPokemon — Unnecessary TreeMap + Redundant Validation

File: App.java, createPokemon() method

Current code:

TreeMap<Integer, String> pokemonOptions = new TreeMap<>();
pokemonOptions.put(1, "Bulbasaur");
// ... 9 entries

if (species == null || !pokemonOptions.containsValue(species)) {
    // containsValue is O(n) scan on a TreeMap
    return null;
}

switch (species) {
    case "Bulbasaur": createdMon = new Bulbasaur(name); break;
    // ... same 9 cases
}

The map is rebuilt every call, only used for validation, and then the switch does the real work. The containsValue() call is O(n) — you gain nothing from the TreeMap.

Better approach — the switch's default branch already handles invalid species, so the map is completely unnecessary:

public static Pokemon createPokemon(String species, String name) {
    if (species == null) {
        System.out.println("Invalid species selected.");
        return null;
    }
    switch (species) {
        case "Bulbasaur":  return new Bulbasaur(name);
        case "Ivysaur":    return new Ivysaur(name);
        case "Venusaur":   return new Venusaur(name);
        case "Charmander": return new Charmander(name);
        case "Charmeleon": return new Charmeleon(name);
        case "Charizard":  return new Charizard(name);
        case "Squirtle":   return new Squirtle(name);
        case "Wartortle":  return new Wartortle(name);
        case "Blastoise":  return new Blastoise(name);
        default:
            System.out.println("Invalid species selected.");
            return null;
    }
}

This eliminates the allocation, population, and scan of the TreeMap entirely.

4. Pokemon.java — EV Setter Doesn't Enforce the 252 Cap

You declare evTotal with a comment saying "This must be 252 or lower," but no setter actually enforces it. Every setEvXxx() method silently accepts any value:

public void setEvAttack(int evAttack) {
    this.evAttack = evAttack;  // no bounds check
}

If you want enforcement, add validation:

public void setEvAttack(int evAttack) {
    if (evAttack < 0 || evAttack > 252) {
        throw new IllegalArgumentException("EV must be 0-252, got: " + evAttack);
    }
    this.evAttack = evAttack;
}

5. Pokémon Species Constructors — Redundant setName() Call

Every species constructor does this:

public Bulbasaur(String name) {
    super("Bulbasaur", 1, "Grass", "Poison", 5, 45, 49, 49, 65, 65, 45);
    this.setName(name);  // ← redundant pattern
}

The super() constructor already sets this.name = species. You are immediately overwriting it with the nickname. This works, but you could add name as a parameter to the Pokemon constructor itself to avoid the extra call in every single subclass.

6. getEvYield() — If/Else Chain That Should Be an Array

File: Pokemon.java

Current code:

public String[] getEvYield() {
    String[] stringYield = new String[evYield.length];
    for (int i = 0; i < evYield.length; i++) {
        if (i == 0) stringYield[i] = "HP: " + evYield[i];
        else if (i == 1) stringYield[i] = "Attack: " + evYield[i];
        // ... 6 branches for fixed indices
    }
    return stringYield;
}

Better approach — a static label array:

private static final String[] STAT_NAMES = {
    "HP", "Attack", "Defense", "Special Attack", "Special Defense", "Speed"
};

public String[] getEvYield() {
    String[] stringYield = new String[evYield.length];
    for (int i = 0; i < evYield.length; i++) {
        stringYield[i] = STAT_NAMES[i] + ": " + evYield[i];
    }
    return stringYield;
}
